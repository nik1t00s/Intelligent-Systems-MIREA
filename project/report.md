# Отчёт по проекту

## Паспорт

- Название: «Сервис прогнозирования риска долгого закрытия Jira issue»
- Домен: управление задачами и issue tracking
- Источник данных: Kaggle `cesaranasco/jira-dataset`
- ML-задача: бинарная классификация `is_delayed`

## Постановка задачи

Цель проекта — помочь project manager или тимлиду заранее выявлять Jira issues, которые с высокой вероятностью будут закрываться дольше обычного. Такой прогноз можно использовать для ранней triage-приоритизации, ручной проверки сложных задач и распределения внимания команды.

Для каждой Jira issue модель получает признаки, доступные на момент создания задачи, и возвращает вероятность попадания issue в группу долгого закрытия.

Целевая переменная построена на основе реальных дат:

```text
resolution_time_days = Resolved - Created
is_delayed = 1, если resolution_time_days > 75-й процентиль
```

В текущей версии проекта 75-й процентиль равен `305.2056` дня. При выборе модели приоритет отдан `F1` и `Recall`, потому что для сценария раннего обнаружения рискованных задач важно не пропустить потенциально проблемные issues.

## Данные

Используется реальный Jira export `data/jira_dataset.csv` с Kaggle. Исходная таблица содержит `49000` строк и большое количество полей Jira export. Для обучения используются только issues с валидными датами `Created` и `Resolved`; после фильтрации остается `16562` строк.

Подготовка данных реализована в `src/data_generation.py`. Подготовленная таблица сохраняется в `artifacts/jira_issues_prepared.csv`.

Признаки модели:

- категориальные: `issue_type`, `priority`;
- бинарные: `has_priority`, `component_present`;
- числовые: `summary_length`, `summary_word_count`, `description_length`, `description_word_count`;
- целевая переменная: `is_delayed`.

Чтобы избежать утечки будущей информации, в модель не передаются `Status`, `Resolution`, `Resolved`, `Updated`, `resolution_time_days` и календарный год создания. Календарные признаки были проверены отдельно, но исключены из финальной модели из-за сильного временного сдвига в датасете.

Шаги предобработки:

- заполнение пропусков категориальных признаков значением `unknown`;
- выделение признаков `summary_length`, `summary_word_count`, `description_length`, `description_word_count`;
- кодирование категорий и масштабирование числовых признаков внутри `ColumnTransformer`.

Графики для отчета генерируются в notebook и сохраняются в `notebooks/figures/*_from_notebook.png`.

Ключевые наблюдения:

- В исходном датасете `49000` Jira issues.
- Дата `Resolved` заполнена примерно у `33.8%` строк.
- После фильтрации решенных issue доступно `16562` объектов.
- Класс `is_delayed = 1` составляет около `24.85%`, то есть задача умеренно несбалансирована.
- Время закрытия имеет длинный хвост: медиана около `124.7` дня, 75-й процентиль около `305.2` дня, максимум более `2500` дней.
- Все решенные issues в выгрузке относятся к проекту `SRCTREEWIN`, поэтому `Project key` не используется как информативный признак.

## Эксперименты

Обучение реализовано в `src/train.py`. Предобработка вынесена в `src/preprocessing.py`: числовые признаки обрабатываются `SimpleImputer + StandardScaler`, категориальные — `SimpleImputer + OneHotEncoder`, бинарные признаки передаются в модель после приведения типов. Использовался пайплайн `ColumnTransformer + classifier`, деление `train/test = 80/20`, `random_state = 42`.

Сравнивались три модели:

| Модель | Accuracy | Precision | Recall | F1 | ROC-AUC |
|---|---:|---:|---:|---:|---:|
| LogisticRegression | 0.5083 | 0.3112 | 0.8068 | 0.4491 | 0.6436 |
| RandomForest | 0.6933 | 0.4465 | 0.9781 | 0.6131 | 0.9198 |
| GradientBoosting | 0.7972 | 0.9364 | 0.1968 | 0.3253 | 0.8960 |

## Выбор финальной модели

Финальной выбрана `RandomForest`, и именно она сериализована в `models/model.joblib` и используется сервисом `/predict`.

Причины выбора:

- лучшая метрика `F1 = 0.6131`;
- лучший `Recall = 0.9781`, что важно для раннего выявления issues с риском долгого закрытия;
- высокий `ROC-AUC = 0.9198`;
- модель лучше улавливает нелинейные связи между длиной описания, типом issue, приоритетом и наличием компоненты.

`GradientBoosting` показал более высокий `Accuracy` и `Precision`, но слишком низкий `Recall = 0.1968`: такая модель часто пропускает долгие issues. Для выбранного бизнес-сценария это хуже.

## Архитектура сервиса

Сервис реализован на FastAPI в `src/service.py`.

Реализованные endpoints:

- `GET /health` — статус приложения, имя активной модели и путь к датасету;
- `POST /predict` — прогноз риска долгого закрытия по списку Jira issues;
- `GET /metrics` — текстовые метрики запросов, ошибок, latency и uptime;
- `GET /docs` — Swagger UI.

Интерфейс Swagger UI работающего сервиса зафиксирован на скриншоте `screenshots/swagger_ui.png` и вставлен в итоговый Word-отчет как рисунок 3.2.

Что делает `/predict`:

1. Валидирует вход через Pydantic-схемы.
2. Преобразует входной JSON в `DataFrame`.
3. Загружает обученную модель из `models/model.joblib`.
4. Возвращает бинарное решение, вероятность, версию модели и текстовый уровень риска.

Формат ответа содержит `model_name`, `model_version` и массив `predictions`, где для каждой issue возвращаются `prediction`, `probability`, `is_delayed`, `delay_probability` и `delay_risk`. Основные статус-коды: `200` для успешного прогноза, `422` для ошибки валидации входного JSON, `400` для некорректных значений на этапе подготовки, `500` для внутренней ошибки предсказания.

Наблюдаемость:

- логирование каждого запроса;
- подсчет количества запросов и ошибок;
- хранение latency последнего запроса;
- endpoint `/health` для проверки готовности сервиса;
- endpoint `/metrics` для базовых метрик.

Конфигурация:

- параметры приложения вынесены в `.env.example`;
- основной train-конфиг лежит в `configs/config.yaml`;
- реальные секреты в проект не добавлялись.

Контейнеризация:

- `Dockerfile` собирает приложение с зависимостями из `requirements.txt`;
- `docker-compose.yml` запускает сервис `jira-delay-risk-service` на порту `8000`;
- команды запуска: `docker build -t jira-delay-risk-service .`, `docker run --rm -p 8000:8000 jira-delay-risk-service`, `docker compose up --build`.

## Ограничения

- Датасет является выгрузкой одного Jira-проекта `SRCTREEWIN`, поэтому переносимость модели на другие проекты требует проверки.
- Целевая переменная построена через относительный порог 75-го процентиля, а не через официальный SLA.
- Текст summary/description используется только через длину и число слов; семантика текста пока не анализируется.
- Сервис не подключен к Jira API и не обновляет модель автоматически.
