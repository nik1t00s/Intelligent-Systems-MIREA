# Отчёт по проекту

## Паспорт

- Название: «Система управления задачами с прогнозированием просрочки задачи»
- Домен: табличный бизнес-сценарий
- Предметная область: управление задачами в небольшой команде
- ML-задача: бинарная классификация `is_overdue`

## Постановка задачи

Цель проекта — помочь тимлиду или project manager заранее находить задачи с высоким риском просрочки и реагировать до фактического срыва срока. Модель получает описание текущего состояния задачи и возвращает вероятность того, что задача будет просрочена.

Практический сценарий использования:

- тимлид просматривает задачи с высоким риском и перераспределяет нагрузку;
- project manager использует прогноз как ранний сигнал для эскалации;
- сервис можно встроить в внутреннюю систему управления задачами.

Ключевая продуктовая идея — лучше заранее обнаружить проблемную задачу, даже ценой умеренного числа ложных срабатываний. Поэтому при выборе модели приоритет отдан `F1` и `Recall`, а не только `Accuracy`.

## Данные

Источник данных — синтетический датасет [data/tasks_synthetic.csv](/n:/AI/Intelligent-Systems-MIREA/project/data/tasks_synthetic.csv), который генерируется скриптом [src/data_generation.py](/n:/AI/Intelligent-Systems-MIREA/project/src/data_generation.py).

В датасете 5000 задач и следующие ключевые признаки:

- категориальные: `priority`, `status`, `task_type`, `sprint_phase`
- числовые: `assignee_experience`, `estimated_hours`, `actual_progress`, `days_since_created`, `comments_count`, `team_size`, `blockers_count`, `recent_reassignments`, `priority_score`, `workload_ratio`
- бинарные: `has_blockers`, `is_customer_facing`, `requires_review`
- целевая переменная: `is_overdue`
- дополнительная человекочитаемая метка риска: `overdue_risk`

Синтетика сделана так, чтобы риск просрочки увеличивался при росте оценки трудоёмкости, числа блокеров, количества переназначений, возраста задачи и нагрузочного коэффициента, а также снижался при большем прогрессе и опыте исполнителя.

## EDA

Основной анализ лежит в [notebooks/01_eda_and_experiments.ipynb](/n:/AI/Intelligent-Systems-MIREA/project/notebooks/01_eda_and_experiments.ipynb).

Ключевые наблюдения:

- Класс умеренно несбалансирован: около `29.9%` задач относятся к просроченным.
- Средняя трудоёмкость задачи составляет `17.52` часа, средний прогресс — `53.37%`.
- Доля просрочки выше у `critical` и `high` задач, чем у `low`.
- Наличие блокеров заметно повышает риск: примерно `35.5%` против `23.1%` у задач без блокеров.
- Наиболее сильные сигналы риска в генерации данных связаны с `days_since_created`, `blockers_count`, `estimated_hours`, `actual_progress`, `priority_score` и `workload_ratio`.

## Эксперименты

Обучение реализовано в [src/train.py](/n:/AI/Intelligent-Systems-MIREA/project/src/train.py).  
Использовался пайплайн `ColumnTransformer + classifier`, деление `train/test = 80/20`, `random_state = 42`.

Сравнивались три модели:

| Модель | Accuracy | Precision | Recall | F1 | ROC-AUC |
|---|---:|---:|---:|---:|---:|
| LogisticRegression | 0.6700 | 0.4637 | 0.6622 | 0.5455 | 0.7270 |
| RandomForest | 0.7190 | 0.5421 | 0.3880 | 0.4522 | 0.6986 |
| GradientBoosting | 0.7320 | 0.6148 | 0.2776 | 0.3825 | 0.6946 |

## Выбор финальной модели

Финальной выбрана `LogisticRegression`, и именно она сериализована в [artifacts/model.joblib](/n:/AI/Intelligent-Systems-MIREA/project/artifacts/model.joblib) и используется сервисом `/predict`.

Причины выбора:

- лучшая метрика `F1 = 0.5455`;
- лучший `Recall = 0.6622`, что важно для раннего выявления проблемных задач;
- лучший `ROC-AUC = 0.7270`;
- простота, интерпретируемость и высокая скорость инференса.

Деревья дали более высокий `accuracy`, но заметно проиграли по полноте. Для данного бизнес-сценария это хуже, потому что сервис начнёт пропускать значимую долю реально просроченных задач.

## Архитектура сервиса

Сервис реализован на [src/service.py](/n:/AI/Intelligent-Systems-MIREA/project/src/service.py) с использованием FastAPI.

Реализованные эндпоинты:

- `GET /health` — статус приложения и имя активной модели
- `POST /predict` — прогноз по списку задач
- `GET /metrics` — текстовые метрики запросов, ошибок, latency и uptime

Что делает `/predict`:

1. Валидирует вход через Pydantic-схемы.
2. Преобразует входной JSON в `DataFrame`.
3. Загружает реальную обученную модель из `artifacts/model.joblib`.
4. Возвращает бинарное решение, вероятность и текстовый уровень риска.

Наблюдаемость:

- логирование каждого запроса;
- подсчёт количества запросов и ошибок;
- хранение latency последнего запроса;
- endpoint `/health` для проверки готовности сервиса.

Конфигурация:

- параметры вынесены в [.env.example](/n:/AI/Intelligent-Systems-MIREA/project/.env.example);
- train-конфиг лежит в [configs/train_config.yaml](/n:/AI/Intelligent-Systems-MIREA/project/configs/train_config.yaml);
- реальные секреты в проект не добавлялись.

## Ограничения

- Данные синтетические и не отражают все реальные организационные паттерны.
- Модель не учитывает текст описания задачи, историю коммитов и реальные дедлайны.
- В сервисе нет базы данных, аутентификации и полноценной системы метрик.
